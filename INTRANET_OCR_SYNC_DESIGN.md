# 内外网代码识别同步工具设计稿

## 1. 目标与约束

这套工具的目标，是把内网设备上的代码，通过采集卡和外网设备的视觉识别能力，尽可能自动地恢复为可维护的代码仓库，并且在后续分支变更时，尽量只处理有变化的部分。

当前约束如下：

- 内网设备是 Linux，环境较老，可用 Python 2.7.16 / 3.7.3，尽量不额外安装软件。
- 外网设备是 Windows / macOS，其中 Windows 机器配置更适合承担 OCR 主链路。
- 外网设备通过 OBS 获取采集卡视频画面。
- 后续希望支持自动化连续执行，而不是人工逐页操作。
- 当内外网分支出现差异时，希望优先做增量更新，而不是全量重跑识图。

设计原则：

- 内网只做“展示”和“变更计算”，不做重识别推理。
- 外网只做“采集、OCR、纠错、落盘、校验”。
- 以固定分页和稳定元数据作为自动化基础。
- 以 `git diff` 结果驱动增量更新。

## 2. 总体架构

系统分为三个逻辑模块：

1. 内网展示端
2. 外网识别端
3. 增量同步控制端

推荐拓扑如下：

```text
内网 Git 仓库
    |
    v
[内网展示器]
  - 计算 branch / commit / diff
  - 读取代码文件
  - 按固定规则分页
  - 在桌面全屏展示
    |
    v
采集卡 -> OBS
    |
    v
[外网识别器]
  - 控制 OBS 抓帧
  - 检测页切换
  - OCR 识别页面
  - 恢复 chunk / file
  - 写入镜像仓库
    |
    v
[同步控制器]
  - 按 commit 建立同步记录
  - 基于 diff 只重跑变更页
  - 生成校验报告
```

## 3. 模块划分

### 3.1 内网展示端

内网展示端建议使用：

- `bash` 负责获取仓库信息和差异信息
- `python3.7` 标准库脚本负责分页显示

职责如下：

- 获取当前仓库根目录、分支名、commit hash
- 生成待展示文件清单
- 根据固定页大小切分代码
- 为每一页生成稳定的页头元数据
- 按顺序自动翻页
- 支持全量展示和增量展示两种模式

不建议直接打开大量编辑器窗口。推荐做一个“专用代码放映器”，原因如下：

- 编辑器主题、换行、缩放、边距不稳定
- OCR 对不规则布局很敏感
- 多窗口切换难以自动化

### 3.2 外网识别端

外网识别端建议优先运行在 Windows 上，原因如下：

- 更适合与 OBS 联动
- PaddleOCR / ONNX Runtime / DirectML 路线更顺手
- 你的硬件更适合本地识别与缓存处理

职责如下：

- 从 OBS 获取实时帧或定时截图
- 识别页头元数据
- 对正文代码区域做 OCR
- 将识别结果归档为 chunk
- 依据 chunk 还原文件
- 进行基础语法纠错和重复页去重
- 写入外网镜像仓库

### 3.3 增量同步控制端

同步控制端可以先与外网识别端写在同一套程序里，后续再拆。

职责如下：

- 记录上一次成功同步的内网 commit
- 根据新旧 commit 计算差异文件
- 将差异行区间映射为页面区间
- 仅请求内网展示变更页
- 在外网侧只更新对应 chunk

## 4. 内网展示协议设计

### 4.1 页面目标

为了提高 OCR 成功率，每一页都应该满足：

- 固定宽度字体
- 固定背景色与文字颜色
- 禁止自动换行
- 固定行高与边距
- 显示绝对文件路径或相对仓库路径
- 显示行号
- 显示页码与 chunk 唯一标识

建议每页展示 40 到 60 行，首版建议固定为 40 行。

### 4.2 页面布局

推荐布局：

```text
============================================================
REPO=spc-cloud
BRANCH=feature/xxx
COMMIT=abc1234567
FILE=spc-modules-system/src/main/java/com/spc/.../lvstConvFileType.java
PAGE=3/9
LINES=81-120
CHUNK=6f0b64a6d4d0
MODE=FULL|DIFF
============================================================
  81  public enum LvstConvFileType {
  82      ...
  83      ...
  ...
 120  }
============================================================
```

其中：

- `FILE` 用于外网落盘定位
- `LINES` 用于 chunk 回写
- `CHUNK` 用于幂等和去重
- `COMMIT` 用于确认来源版本

### 4.3 chunk_id 生成

建议以以下字段拼接后做哈希：

- 相对路径
- 页起始行号
- 页结束行号
- 页正文原文

示例：

```text
chunk_id = sha1(path + "\n" + start_line + "\n" + end_line + "\n" + body_text)
```

作用：

- 外网识别到相同 chunk 时可直接跳过
- 可判断当前页是否真正变化
- 可避免重复 OCR 与重复写文件

### 4.4 翻页方式

建议支持两种运行模式：

- 自动播放模式
- 手动步进模式

自动播放模式建议参数化：

- `--page-lines 40`
- `--dwell-ms 1800`
- `--mode full`
- `--mode diff`

建议在每次开始与结束时增加“状态页”：

- 开始页：显示仓库、分支、commit、总页数
- 结束页：显示本轮结束和摘要信息

这样外网识别端更容易判断批次边界。

## 5. 外网识别流程设计

### 5.1 OBS 集成

外网设备通过 OBS 获取采集卡画面，推荐使用 `obs-websocket` 自动控制。

推荐流程：

1. OBS 打开指定场景和采集源
2. 识别程序通过 WebSocket 定时请求截图
3. 每隔固定时间获取一张新图
4. 对页头区域先做轻量识别
5. 若 `CHUNK` 未变化，则跳过正文 OCR
6. 若 `CHUNK` 变化，则识别正文并写缓存

这样能显著减少重复 OCR。

### 5.2 页面识别分层

建议将识别分成两层：

1. 元数据识别层
2. 代码正文识别层

元数据识别层只识别页头，用于快速确定：

- 当前是否新页
- 当前页对应哪个文件
- 当前页属于哪个 commit
- 当前页对应哪一段行号

正文识别层只处理代码区域。

### 5.3 OCR 模型建议

首版建议：

- OCR：PaddleOCR
- 图像预处理：灰度化、二值化、裁切、锐化
- 后处理：基于语言规则的纠错器

第二阶段再考虑：

- ONNX Runtime
- DirectML
- AMD Ryzen AI / NPU 适配

建议先不要把 NPU 作为首版前提，因为首版最重要的是跑通：

- 页面定位
- 元数据提取
- 增量同步
- chunk 级写回

### 5.4 代码后处理

OCR 后不要直接整页写文件，建议增加“代码恢复器”。

恢复器至少要做以下事情：

- 去掉页头页尾分隔线
- 识别行号并剥离
- 恢复缩进
- 修正常见误识别字符
- 对多页内容进行顺序拼接
- 在写入前做基础语法检查

常见误识别样例：

- `O` 与 `0`
- `I` 与 `1`
- `l` 与 `1`
- `;` 与 `:`
- `{` 与 `(`
- `}` 与 `)`
- `,` 与 `.`

不同语言可以做不同修复器：

- Java 修复器
- Python 修复器
- JavaScript / TypeScript 修复器

首版可先只支持 Java。

## 6. 增量同步设计

### 6.1 增量的核心思路

不建议“只要分支不同就全量重跑”。应该使用以下三级增量：

1. 文件级增量
2. 行块级增量
3. 页面级增量

### 6.2 文件级增量

内网通过以下命令计算文件差异：

```bash
git diff --name-status <old_commit> <new_commit>
```

可获得：

- `A` 新增文件
- `M` 修改文件
- `D` 删除文件
- `R` 重命名文件

处理建议：

- `A`：新文件按全量分页展示
- `M`：继续进入行块级计算
- `D`：向外网发送删除指令
- `R`：尽量保留旧文件内容并做路径迁移

### 6.3 行块级增量

对修改文件，使用：

```bash
git diff -U0 <old_commit> <new_commit> -- <path>
```

作用：

- 只输出有变化的行区间
- 更适合映射页面

例如：

- 文件固定每页 40 行
- 变更行是 85-103
- 则只需重跑第 3 页

如果变更跨页，则重跑相邻多个页。

### 6.4 页面级增量

将行区间映射成页面范围：

```text
page_no = (line_no - 1) // page_lines + 1
```

将所有变化行映射后取并集，最终得到待重跑页面列表。

示例：

- 变更区间：10-18, 35-52, 105-110
- 每页 40 行
- 受影响页：1, 2, 3

### 6.5 外网侧幂等更新

外网收到新的 chunk 后，按如下策略处理：

- 如果 `chunk_id` 已存在且内容相同，则跳过
- 如果 `chunk_id` 不同但 `FILE + LINES` 相同，则替换对应区间
- 如果文件已被删除，则从镜像仓库移除
- 如果文件重命名，则同步更新路径索引

这样就能在大量文件不变时，避免整仓重新识别。

## 7. 自动化执行流程

### 7.1 首次全量同步

1. 内网侧选择目标仓库
2. 读取当前分支和 commit
3. 生成代码文件清单
4. 对所有文件分页
5. 全屏轮播所有页面
6. 外网侧抓取每一页并 OCR
7. 外网侧写入镜像仓库
8. 记录同步元数据：分支、commit、chunk 清单

### 7.2 日常增量同步

1. 内网侧读取上次已同步 commit
2. 内网侧执行 `git diff`
3. 算出变更文件与变更页
4. 仅轮播变更页
5. 外网侧仅处理新 `chunk_id`
6. 合并更新镜像仓库文件
7. 完成后记录新的同步基线

### 7.3 失败恢复

建议记录以下状态文件：

- `last_synced_commit.json`
- `chunk_index.json`
- `page_capture_log.json`
- `rebuild_report.json`

这样当中断后可以从上次成功页继续，而不是从头开始。

## 8. 数据文件设计

建议在外网侧建立如下目录：

```text
mirror_repo/
capture_cache/
ocr_raw/
ocr_fixed/
manifests/
reports/
```

建议文件说明：

- `mirror_repo/`：外网还原后的代码镜像仓库
- `capture_cache/`：OBS 截图缓存
- `ocr_raw/`：未经修正的 OCR 结果
- `ocr_fixed/`：纠错后的 chunk 内容
- `manifests/`：commit、chunk、文件映射信息
- `reports/`：同步报告和异常报告

建议 manifest 结构：

```json
{
  "repo": "spc-cloud",
  "branch": "feature/demo",
  "commit": "abc1234",
  "page_lines": 40,
  "chunks": [
    {
      "file": "spc-modules-system/src/main/java/.../lvstConvFileType.java",
      "start_line": 81,
      "end_line": 120,
      "page": 3,
      "chunk_id": "6f0b64a6d4d0",
      "image": "capture_cache/abc1234_00023.png",
      "status": "ok"
    }
  ]
}
```

## 9. 首版实现建议

为了降低风险，建议先做 MVP，不要一开始追求全语言、全平台、全自动。

MVP 范围建议如下：

- 内网只支持 Linux
- 外网只支持 Windows
- 代码类型先只支持 `.java`
- 页面固定 40 行
- 用 Python 3.7 标准库实现内网展示器
- 用 Python 3.x 实现外网识别器
- 用 OBS + `obs-websocket` 完成自动截图
- 用 PaddleOCR 完成正文识别
- 先支持全量同步和基于 `git diff` 的增量同步
- 先不接 NPU，后续再优化加速

## 10. 推荐的脚本拆分

建议后续落地为以下脚本：

### 内网侧

- `intranet/list_files.sh`
  - 生成候选代码文件清单
- `intranet/compute_diff.sh`
  - 计算新旧 commit 差异
- `intranet/render_pages.py`
  - 分页并全屏显示代码页面
- `intranet/run_sync.sh`
  - 一键执行全量或增量展示

### 外网侧

- `external/obs_capture.py`
  - 从 OBS 获取截图
- `external/page_detector.py`
  - 判断是否切换到新页
- `external/ocr_runner.py`
  - 执行 OCR
- `external/code_rebuilder.py`
  - chunk 拼接与文件恢复
- `external/sync_manager.py`
  - 同步控制、状态记录与异常恢复

## 11. 关键风险与规避建议

### 11.1 OCR 准确率风险

风险：

- 小字体、低对比度、压缩噪声、滚动模糊都会降低识别率。

规避建议：

- 大字号等宽字体
- 明暗对比强
- 固定截图分辨率
- 页面切换后延迟截图
- 只识别裁剪后的正文区域

### 11.2 页边界不稳定风险

风险：

- 如果窗口尺寸变化，同一文件可能被分成不同页。

规避建议：

- 每页固定行数
- 页面尺寸固定
- 禁止自动换行
- chunk 计算与页切分完全解耦于窗口真实像素高度

### 11.3 重命名与删除风险

风险：

- 只做 OCR 更新时，无法自然感知文件删除或 rename。

规避建议：

- 必须以 `git diff --name-status` 作为真相来源
- 删除和 rename 不依赖 OCR 推断

### 11.4 大仓库性能风险

风险：

- 文件过多时，全量分页和 OCR 时间会很长。

规避建议：

- 首次全量后只做增量
- 跳过第三方依赖目录
- 按语言和目录配置白名单
- 通过 `chunk_id` 跳过重复页

## 12. 下一步建议

建议按以下顺序开发：

1. 先写内网分页展示器
2. 再写外网的截图与元数据识别
3. 再打通 chunk 缓存和文件回写
4. 最后再接入 `git diff` 增量同步
