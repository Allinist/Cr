# 内外网代码识别同步工具设计稿

## 1. 目标与约束

这套工具的目标，是把内网设备上的代码通过采集卡和外网设备的视觉识别能力，尽可能自动地恢复为可维护的代码仓库，并且在后续分支变更时，尽量只处理有变化的部分。

当前约束如下：

- 内网设备是 Linux，环境较老，可用 Python 2.7.16 / 3.7.3，尽量不额外安装软件。
- 外网设备是 Windows / macOS，其中 Windows 机器更适合承担 OCR 主链路。
- 外网设备通过 OBS 获取采集卡视频画面。
- 后续希望支持自动化连续执行，而不是人工逐页操作。
- 当内外网分支出现差异时，希望优先做增量更新，而不是全量重跑识图。
- 代码栈以 Spring Boot + MyBatis 为主，需要重点覆盖 `.java`、`.xml`、`.sql`、`.properties`、`.yml` 等文件。
- 对 `.class` 文件不能按源码文件同等对待，需要单独定义策略。

设计原则：

- 内网只做“展示”和“变更计算”，不做重识别推理。
- 外网只做“采集、OCR、纠错、落盘、校验”。
- 以固定分页和稳定元数据作为自动化基础。
- 以 `git diff` 结果驱动增量更新。
- 任何分页策略都必须保证“内容可穷尽恢复”，不能因为长行、超宽 XML、长 SQL 而漏字漏段。

## 2. 总体架构

系统分为三个逻辑模块：

1. 内网展示端
2. 外网识别端
3. 增量同步控制端

推荐拓扑如下：

```text
内网 Git 仓库
    |
    v
[内网展示器]
  - 计算 branch / commit / diff
  - 读取代码文件
  - 按固定规则分页 / 分片
  - 在桌面全屏展示
    |
    v
采集卡 -> OBS
    |
    v
[外网识别器]
  - 控制 OBS 抓帧
  - 检测页切换
  - OCR 识别页面
  - 恢复 chunk / file
  - 写入镜像仓库
    |
    v
[同步控制器]
  - 按 commit 建立同步记录
  - 基于 diff 只重跑变更页
  - 生成校验报告
```

## 3. 文件类型支持范围

### 3.1 首要支持的源码与配置文件

建议第一优先级支持以下文件：

- `.java`
- `.xml`
- `.sql`
- `.properties`
- `.yml`
- `.yaml`
- `.sh`
- `.md`
- `.txt`

这是因为 Spring Boot + MyBatis 项目里，核心逻辑通常分散在：

- Java 业务代码
- MyBatis Mapper XML
- 初始化 SQL / 变更 SQL
- Spring Boot 配置文件
- Shell 启动和部署脚本

### 3.2 `.class` 文件的处理建议

`.class` 文件不建议纳入 OCR 恢复主链路，原因如下：

- `.class` 是二进制字节码，不适合文本 OCR。
- 即使显示十六进制或反编译文本，准确性也很难保证。
- 外网恢复目标通常是“可编辑源码仓库”，而不是字节码镜像。

因此建议分成两种策略：

1. 默认忽略 `.class`
2. 如必须保留，使用“存在性索引 + 哈希摘要”模式

也就是说：

- 内网不展示 `.class` 正文
- 只展示或导出它的路径、大小、修改时间、哈希
- 外网镜像中记录为一个索引清单，例如 `binary_manifest.json`

如果将来确实需要分析 `.class`，建议走“外网反编译辅助链路”，但这不属于 OCR 主链路的一部分。

## 4. 模块划分

### 4.1 内网展示端

内网展示端建议使用：

- `bash` 负责获取仓库信息和差异信息
- `python3.7` 标准库脚本负责分页显示

职责如下：

- 获取当前仓库根目录、分支名、commit hash
- 生成待展示文件清单
- 根据固定页大小切分代码
- 对超长行做横向切片
- 为每一页生成稳定的页头元数据
- 按顺序自动翻页
- 支持全量展示和增量展示两种模式

不建议直接打开大量编辑器窗口。推荐做一个“专用代码放映器”，原因如下：

- 编辑器主题、换行、缩放、边距不稳定
- OCR 对不规则布局很敏感
- 多窗口切换难以自动化

### 4.2 外网识别端

外网识别端建议优先运行在 Windows 上，原因如下：

- 更适合与 OBS 联动
- PaddleOCR / ONNX Runtime / DirectML 路线更顺手
- 你的硬件更适合本地识别与缓存处理

职责如下：

- 从 OBS 获取实时帧或定时截图
- 识别页头元数据
- 对正文代码区域做 OCR
- 将识别结果归档为 chunk
- 依据 chunk 还原文件
- 进行基础语法纠错和重复页去重
- 写入外网镜像仓库

### 4.3 增量同步控制端

同步控制端可以先与外网识别端写在同一套程序里，后续再拆。

职责如下：

- 记录上一次成功同步的内网 commit
- 根据新旧 commit 计算差异文件
- 将差异行区间映射为页面区间
- 仅请求内网展示变更页
- 在外网侧只更新对应 chunk

## 5. 内网展示协议设计

### 5.1 页面目标

为了提高 OCR 成功率，每一页都应该满足：

- 固定宽度字体
- 固定背景色与文字颜色
- 禁止自动换行
- 固定行高与边距
- 显示绝对文件路径或相对仓库路径
- 显示行号
- 显示页码与 chunk 唯一标识

建议每页展示 40 到 60 行，首版建议固定为 40 行。

### 5.2 页面布局

推荐布局：

```text
====================================================================
REPO=spc-cloud
BRANCH=feature/xxx
COMMIT=abc1234567
FILE=src/main/resources/mapper/UserMapper.xml
LANG=xml
PAGE=3/9
LINES=81-120
SLICE=1/2
VISUAL_MODE=page+slice
CHUNK=6f0b64a6d4d0
====================================================================
  81  <select id="queryUserPage" resultMap="BaseResultMap">
  82      SELECT u.id, u.user_name, u.status, ...
  83      ...
 120  </select>
====================================================================
```

其中：

- `FILE` 用于外网落盘定位
- `LANG` 用于选择不同纠错器
- `LINES` 用于 chunk 回写
- `SLICE` 用于超长行横向切片
- `CHUNK` 用于幂等和去重
- `COMMIT` 用于确认来源版本

### 5.3 chunk_id 生成

建议以以下字段拼接后做哈希：

- 相对路径
- 页起始行号
- 页结束行号
- 横向切片编号
- 页正文原文

示例：

```text
chunk_id = sha1(path + "\n" + start_line + "\n" + end_line + "\n" + slice_no + "\n" + body_text)
```

作用：

- 外网识别到相同 chunk 时可直接跳过
- 可判断当前页是否真正变化
- 可避免重复 OCR 与重复写文件

### 5.4 翻页方式

建议支持两种运行模式：

- 自动播放模式
- 手动步进模式

自动播放模式建议参数化：

- `--page-lines 40`
- `--page-cols 110`
- `--dwell-ms 1800`
- `--mode full`
- `--mode diff`

建议在每次开始与结束时增加“状态页”：

- 开始页：显示仓库、分支、commit、总页数
- 结束页：显示本轮结束和摘要信息

这样外网识别端更容易判断批次边界。

## 6. 长行与不漏代码策略

这是这套系统里最关键的补充点。对于 Spring Boot + MyBatis 项目，长行问题非常常见，尤其集中在：

- MyBatis XML 里的长 SQL
- Java 里的长注解、长字符串、链式调用
- `application.yml` 的长配置值
- `.properties` 中的 URL、连接串
- 单行 SQL 文件

如果只按“固定行数分页”而不处理“超宽行”，一定会漏内容。因此建议采用“纵向分页 + 横向切片”的双维度模型。

### 6.1 纵向分页

纵向分页负责处理“多少行”：

- 每页固定 40 行
- 页号由 `LINES` 标记
- 这一层解决上下方向的内容完整性

### 6.2 横向切片

横向切片负责处理“单行太长”：

- 每页设定可视列宽，例如 110 列
- 如果某一行超出 110 列，则不换行，而是按列区间切成多个 slice
- 同一页内容需要按 `SLICE=1/3, 2/3, 3/3` 依次展示

示意如下：

```text
PAGE=5/12
LINES=161-200
SLICE=1/3
COLS=1-110
```

```text
PAGE=5/12
LINES=161-200
SLICE=2/3
COLS=111-220
```

```text
PAGE=5/12
LINES=161-200
SLICE=3/3
COLS=221-330
```

这样做的好处是：

- 不依赖自动换行
- OCR 每次看到的布局稳定
- 外网可以按 `PAGE + SLICE` 重建完整文本

### 6.3 为什么不建议自动换行

自动换行虽然看起来省事，但会带来严重问题：

- 同一行在不同分辨率下换行位置不同
- OCR 难以判断一行何时结束
- 外网难以区分“原始换行”和“视觉换行”
- diff 无法稳定映射回原始列范围

所以建议坚持：

- 展示器内部绝不自动换行
- 只做显式横向切片
- 每个 slice 都带明确列范围

### 6.4 重建规则

外网重建文件时按以下顺序恢复：

1. 先按 `FILE`
2. 再按 `LINES`
3. 再按 `SLICE`
4. 最后按 `COLS`

同一行如果存在多个 slice，则按列顺序拼接后恢复成完整原文。

### 6.5 不漏内容的校验机制

建议增加三层校验：

1. 页级校验
2. 行级校验
3. 文件级校验

页级校验：

- 检查某个 `PAGE + SLICE` 是否全部到齐

行级校验：

- 检查 `LINES` 区间内每一行是否都出现
- 若某行存在横向切片，检查该行列范围是否闭合无缺口

文件级校验：

- 根据内网生成的 `line_count`、`max_visual_cols`、`page_count`、`slice_count` 校验外网重建结果是否完整

建议内网在每个文件结束时额外展示一个“文件摘要页”，内容包括：

- `TOTAL_LINES`
- `MAX_LINE_LEN`
- `TOTAL_PAGES`
- `TOTAL_SLICES`
- `FILE_SHA1`

外网重建完后据此做完整性校验。

## 7. Spring Boot + MyBatis 专项策略

### 7.1 `.java`

特点：

- 语法规则强
- 可通过括号、分号、关键字做后处理纠错

建议：

- 保留行号
- 优先做语法型纠错
- 后续可接 Java parser 做基础校验

### 7.2 `.xml`

特点：

- 标签、属性、转义符多
- MyBatis 中常混合 XML + SQL + 动态标签

建议：

- 不能自动折行
- 优先展示原始缩进
- 后处理时增加 XML 标签闭合校验
- 针对 MyBatis 标签增加专项规则：
  - `<if>`
  - `<choose>`
  - `<when>`
  - `<otherwise>`
  - `<trim>`
  - `<where>`
  - `<set>`
  - `<foreach>`

### 7.3 `.sql`

特点：

- 长行比例高
- 关键字稳定
- 表名、字段名、别名容易被 OCR 误判

建议：

- SQL 文件默认启用横向切片
- 对 `SELECT`、`FROM`、`WHERE`、`LEFT JOIN` 等关键字做字典纠错
- 如是 XML 中嵌入 SQL，则按 XML 文件统一处理，不拆双重模式

### 7.4 `.properties` / `.yml`

特点：

- 经常有长 URL、长 token、连接串
- 冒号、等号、点号、短横线非常密集

建议：

- 配置文件也启用横向切片
- 强化 `:`、`=`、`.`、`-` 的识别纠错
- 对缩进层级做校验，尤其是 YAML

### 7.5 `.class`

建议：

- 不进入 OCR 恢复主流程
- 只进入二进制索引流程

## 8. 外网识别流程设计

### 8.1 OBS 集成

外网设备通过 OBS 获取采集卡画面，推荐使用 `obs-websocket` 自动控制。

推荐流程：

1. OBS 打开指定场景和采集源
2. 识别程序通过 WebSocket 定时请求截图
3. 每隔固定时间获取一张新图
4. 对页头区域先做轻量识别
5. 若 `CHUNK` 未变化，则跳过正文 OCR
6. 若 `CHUNK` 变化，则识别正文并写缓存

这样能显著减少重复 OCR。

### 8.2 页面识别分层

建议将识别分成两层：

1. 元数据识别层
2. 代码正文识别层

元数据识别层只识别页头，用于快速确定：

- 当前是否新页
- 当前页对应哪个文件
- 当前页属于哪个 commit
- 当前页对应哪一段行号
- 当前页属于哪个横向 slice

正文识别层只处理代码区域。

### 8.3 OCR 模型建议

首版建议：

- OCR：PaddleOCR
- 图像预处理：灰度化、二值化、裁切、锐化
- 后处理：基于语言规则的纠错器

第二阶段再考虑：

- ONNX Runtime
- DirectML
- AMD Ryzen AI / NPU 适配

建议先不要把 NPU 作为首版前提，因为首版最重要的是跑通：

- 页面定位
- 元数据提取
- 增量同步
- chunk 级写回

### 8.4 代码后处理

OCR 后不要直接整页写文件，建议增加“代码恢复器”。

恢复器至少要做以下事情：

- 去掉页头页尾分隔线
- 识别行号并剥离
- 恢复缩进
- 修正常见误识别字符
- 对多页多 slice 内容进行顺序拼接
- 在写入前做基础语法检查

常见误识别样例：

- `O` 与 `0`
- `I` 与 `1`
- `l` 与 `1`
- `;` 与 `:`
- `{` 与 `(`
- `}` 与 `)`
- `,` 与 `.`
- `<` 与 `(`
- `>` 与 `)`
- `"` 与 `'`

不同语言可以做不同修复器：

- Java 修复器
- XML 修复器
- SQL 修复器
- YAML / Properties 修复器

## 9. 增量同步设计

### 9.1 增量的核心思路

不建议“只要分支不同就全量重跑”。应该使用以下三级增量：

1. 文件级增量
2. 行块级增量
3. 页面级增量

### 9.2 文件级增量

内网通过以下命令计算文件差异：

```bash
git diff --name-status <old_commit> <new_commit>
```

可获得：

- `A` 新增文件
- `M` 修改文件
- `D` 删除文件
- `R` 重命名文件

处理建议：

- `A`：新文件按全量分页展示
- `M`：继续进入行块级计算
- `D`：向外网发送删除指令
- `R`：尽量保留旧文件内容并做路径迁移

### 9.3 行块级增量

对修改文件，使用：

```bash
git diff -U0 <old_commit> <new_commit> -- <path>
```

作用：

- 只输出有变化的行区间
- 更适合映射页面

例如：

- 文件固定每页 40 行
- 变更行是 85-103
- 则只需重跑第 3 页

如果变更跨页，则重跑相邻多个页。

### 9.4 长行场景下的增量

当文件存在超长行时，除了页号，还要计算切片号。

建议：

- 以“原始文件内容”为准计算每行视觉切片数
- 变更落在哪几行，就重算这些行所在页的全部 slice
- 不建议只重跑单一 slice 的局部列范围，首版太复杂，容易丢内容

也就是说，首版增量粒度建议到：

- `file + page`

而不是：

- `file + page + partial-columns`

这样更稳。

### 9.5 页面级增量

将行区间映射成页面范围：

```text
page_no = (line_no - 1) // page_lines + 1
```

将所有变化行映射后取并集，最终得到待重跑页面列表。

如该页存在横向切片，则该页的全部 slice 一并重跑。

### 9.6 外网侧幂等更新

外网收到新的 chunk 后，按如下策略处理：

- 如果 `chunk_id` 已存在且内容相同，则跳过
- 如果 `chunk_id` 不同但 `FILE + LINES + SLICE` 相同，则替换对应区间
- 如果文件已被删除，则从镜像仓库移除
- 如果文件重命名，则同步更新路径索引

这样就能在大量文件不变时，避免整仓重新识别。

## 10. 自动化执行流程

### 10.1 首次全量同步

1. 内网侧选择目标仓库
2. 读取当前分支和 commit
3. 生成代码文件清单
4. 对所有文件分页与切片
5. 全屏轮播所有页面
6. 外网侧抓取每一页并 OCR
7. 外网侧写入镜像仓库
8. 记录同步元数据：分支、commit、chunk 清单

### 10.2 日常增量同步

1. 内网侧读取上次已同步 commit
2. 内网侧执行 `git diff`
3. 算出变更文件与变更页
4. 仅轮播变更页及其全部 slice
5. 外网侧仅处理新 `chunk_id`
6. 合并更新镜像仓库文件
7. 完成后记录新的同步基线

### 10.3 失败恢复

建议记录以下状态文件：

- `last_synced_commit.json`
- `chunk_index.json`
- `page_capture_log.json`
- `rebuild_report.json`

这样当中断后可以从上次成功页继续，而不是从头开始。

## 11. 数据文件设计

建议在外网侧建立如下目录：

```text
mirror_repo/
capture_cache/
ocr_raw/
ocr_fixed/
manifests/
reports/
```

建议文件说明：

- `mirror_repo/`：外网还原后的代码镜像仓库
- `capture_cache/`：OBS 截图缓存
- `ocr_raw/`：未经修正的 OCR 结果
- `ocr_fixed/`：纠错后的 chunk 内容
- `manifests/`：commit、chunk、文件映射信息
- `reports/`：同步报告和异常报告

建议 manifest 结构：

```json
{
  "repo": "spc-cloud",
  "branch": "feature/demo",
  "commit": "abc1234",
  "page_lines": 40,
  "page_cols": 110,
  "chunks": [
    {
      "file": "src/main/resources/mapper/UserMapper.xml",
      "lang": "xml",
      "start_line": 81,
      "end_line": 120,
      "page": 3,
      "slice_no": 1,
      "slice_total": 2,
      "col_start": 1,
      "col_end": 110,
      "chunk_id": "6f0b64a6d4d0",
      "image": "capture_cache/abc1234_00023.png",
      "status": "ok"
    }
  ]
}
```

## 12. 首版实现建议

为了降低风险，建议先做 MVP，不要一开始追求全语言、全平台、全自动。

MVP 范围建议如下：

- 内网只支持 Linux
- 外网只支持 Windows
- 优先支持 `.java`、`.xml`、`.sql`、`.properties`、`.yml`
- `.class` 先只做索引，不做 OCR 恢复
- 页面固定 40 行
- 默认可视宽度固定 110 列
- 对超长行启用横向切片
- 用 Python 3.7 标准库实现内网展示器
- 用 Python 3.x 实现外网识别器
- 用 OBS + `obs-websocket` 完成自动截图
- 用 PaddleOCR 完成正文识别
- 先支持全量同步和基于 `git diff` 的增量同步
- 先不接 NPU，后续再优化加速

## 14. 关键风险与规避建议

### 14.1 OCR 准确率风险

风险：

- 小字体、低对比度、压缩噪声、滚动模糊都会降低识别率。

规避建议：

- 大字号等宽字体
- 明暗对比强
- 固定截图分辨率
- 页面切换后延迟截图
- 只识别裁剪后的正文区域

### 14.2 页边界不稳定风险

风险：

- 如果窗口尺寸变化，同一文件可能被分成不同页。

规避建议：

- 每页固定行数
- 每页固定列宽
- 页面尺寸固定
- 禁止自动换行
- chunk 计算与页切分完全解耦于窗口真实像素高度

### 14.3 XML / SQL 超宽内容漏识别风险

风险：

- 只按行分页时，长 SQL、长 XML 属性会被截断。

规避建议：

- 启用横向切片
- 每个 slice 带列范围
- 外网按列范围重建
- 文件结束后做 `FILE_SHA1` 完整性比对

### 14.4 重命名与删除风险

风险：

- 只做 OCR 更新时，无法自然感知文件删除或 rename。

规避建议：

- 必须以 `git diff --name-status` 作为真相来源
- 删除和 rename 不依赖 OCR 推断

### 14.5 大仓库性能风险

风险：

- 文件过多时，全量分页和 OCR 时间会很长。

规避建议：

- 首次全量后只做增量
- 跳过第三方依赖目录
- 按语言和目录配置白名单
- 通过 `chunk_id` 跳过重复页
